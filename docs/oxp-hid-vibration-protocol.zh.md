# OXP HID 震动协议（CID 0xB3）

通过 Windows USB 抓包在 X1 Mini（CH32 MCU，VID: `0x1A86`，PID: `0xFE00`）上逆向分析。

## 概述

OXP vendor HID 接口通过命令 ID `0xB3` 支持震动控制。与按键映射协议（CID `0xB4`）
不同，震动命令使用**读写配对**协议：每次震动更新需要先读取当前状态，再写入新状态，
且写入数据中包含当前值和上一次的值。

## 数据帧格式

震动命令使用与按键映射（`0xB4`）相同的 `0x3F` 帧格式，CID 为 `0xB3`，双页结构。

### 包结构

```
[CID] [0x3F] [0x01] [页码] [...数据...] [0x3F] [CID]
 0xB3                                            0xB3
```

每次震动更新需要 **4 个包**，按顺序发送：

| 步骤 | 页码 | byte5 | 用途 |
|------|------|-------|------|
| 1 | 第一页 | `0x02` | 读取当前状态（查询） |
| 2 | 第一页 | `0x20` | 写入状态（包含当前值） |
| 3 | 第二页 | `0x02` | 读取当前状态（查询） |
| 4 | 第二页 | `0x20` | 写入新的震动值 |

- `byte5 = 0x02`：查询/读取操作
- `byte5 = 0x20`：修改/写入操作

## 第一页 — 状态回读

第一页用于读回和确认当前震动状态。

### 第一页读取（步骤 1）

```
B3 3F 01 01 00 02 00 00 ... 00 00 00 00 3F B3
                ^  ^
              页码 byte5=读取
```

数据区全部为零 — 这是一个纯读取请求。

### 第一页写入（步骤 2）

```
B3 3F 01 01 00 20 E3 39 E3 39 E3 39 E3 39 E3 [mode] [left] [right] 00 05 ...
              ^                                 b15    b16    b17
            byte5=写入
```

| 字节 | 字段 | 说明 |
|------|------|------|
| 6-14 | 常量 | `E3 39 E3 39 E3 39 E3 39 E3`（固定模式） |
| 15 | 模式 | 当前模式：`0x01`=激活，`0x02`=关闭 |
| 16 | 左马达 | 当前左马达强度 (0-5) |
| 17 | 右马达 | 当前右马达强度 (0-5) |

## 第二页 — 震动控制

第二页携带实际的震动命令，包含新值和上一次的值。

### 第二页读取（步骤 3）

```
B3 3F 01 02 38 02 E3 39 E3 39 E3 39 [new_mode] [new_L] [new_R] [prev_mode] [prev_L] [prev_R] 00 05 ...
              ^                        b12        b13     b14      b15        b16       b17
            byte5=读取
```

### 第二页写入（步骤 4）

```
B3 3F 01 02 38 20 E3 39 E3 39 E3 39 [new_mode] [new_L] [new_R] [prev_mode] [prev_L] [prev_R] 00 05 ...
              ^                        b12        b13     b14      b15        b16       b17
            byte5=写入
```

| 字节 | 字段 | 说明 |
|------|------|------|
| 6-11 | 常量 | `E3 39 E3 39 E3 39`（固定模式） |
| 12 | 新模式 | `0x01`=激活（非零强度），`0x02`=关闭（零强度） |
| 13 | 新左马达 | 新的左马达强度 (0-5) |
| 14 | 新右马达 | 新的右马达强度 (0-5) |
| 15 | 上次模式 | 上一次的模式值 |
| 16 | 上次左马达 | 上一次的左马达强度 |
| 17 | 上次右马达 | 上一次的右马达强度 |
| 19 | 未知 | 始终为 `0x05` |

## 强度等级

马达强度使用 0-5 等级：

| 值 | 强度 |
|----|------|
| 0 | 关闭 |
| 1 | 最小 |
| 2 | 低 |
| 3 | 中 |
| 4 | 高 |
| 5 | 最大 |

## 观察到的行为

在 Windows 上，驱动程序在初始化时会让控制器震动一次，可能是为了确认震动硬件正常。

### 抓包数据（手动扫描）

以下数据通过在 Windows 下手动将震动强度从 0 调到 5 再调回 0 来抓取，
以获得足够多的数据包进行协议分析：

```
步骤  模式  左马达  右马达  （上次值）
 0    开启    1      1     ← 从 关闭 0,0
 1    开启    2      2     ← 从 开启 1,1
 2    开启    3      3     ← 从 开启 2,2
 3    开启    4      4     ← 从 开启 3,3
 4    开启    5      5     ← 从 开启 4,4（峰值）
 5    开启    4      4     ← 从 开启 5,5
 6    开启    3      3     ← 从 开启 4,4
 7    开启    2      2     ← 从 开启 3,3
 8    开启    1      1     ← 从 开启 2,2
 9    关闭    0      0     ← 从 开启 1,1
```

每一步需要完整的 4 包读写循环（第一页读 → 第一页写 → 第二页读 → 第二页写），
10 步扫描共计 **40 个数据包**。

## 与 B4（按键映射）协议对比

| 特性 | B4（按键映射） | B3（震动） |
|------|--------------|-----------|
| CID | `0xB4` | `0xB3` |
| 帧格式 | `0x3F` | `0x3F` |
| 页数 | 2（仅写入） | 2（读写配对） |
| 每次更新包数 | 2 | 4 |
| 第二页 byte4-5 | `0x38 0x20` | `0x38 0x02`/`0x38 0x20` |
| 状态跟踪 | 无 | 有（携带上次值） |
| 持久性 | 固件 flash | 仅当前会话 |

## 实现状态

Vendor 震动（B3）在 InputPlumber 中**未实现**。Xbox 手柄的原生震动通过标准 XInput
路径独立工作。B3 协议之前曾尝试实现，但在非拦截模式下会导致 Xbox 手柄静默，因此已移除。

读写配对协议（需要状态跟踪和每次更新 4 个包）比 B4 的"发完即忘"方式复杂得多。
正确的实现需要：

1. 跟踪上一次的马达值
2. 每次更新执行 第一页读 → 第一页写 → 第二页读 → 第二页写
3. 可能需要在初始化时运行类似 Windows 的校准流程
