# OXP HID 按键映射协议

来源：`GD32--CH32 Mapping analysis26.1.15.xlsx`（壹号本内部文档）

本文档描述 OXP 掌机设备使用 CH32 单片机（HID 通信，VID: `0x1A86`，PID: `0xFE00`）的
按键映射协议。GD32 变体用于更早期的设备（串口通信），仅 CH32（HID）协议与 InputPlumber
相关。

## 概述

映射系统允许将任意物理按键重映射为不同的手柄按键、键盘按键、宏、连发功能或第二功能
（Fn）。配置通过 vendor HID 接口（usage page `0xFF00`）使用命令 ID `0xB4` 发送。

## 数据帧格式

一次完整的映射更新需要发送**两包数据**（每包 64 字节）。单片机在识别到第二包时触发
底层更新和保存。

### 包结构

```
[CID] [0x3F] [0x01] [查询/修改] [数据长度] [协议常量] [页码] [主/副] [按键模块数据...] [0x3F] [CID]
 b1     b2     b3      b4         b5         b6       b7     b8      b9-b62            b63    b64
```

| 字节 | 字段 | 说明 |
|------|------|------|
| 1 | CID | 命令 ID：按键映射为 `0xB4` |
| 2 | 帧头 | 固定：`0x3F` |
| 3 | 内置接收器标志 | 固定：`0x01` |
| 4 | 查询/修改 | `0x01` = 查询当前配置，`0x02` = 修改配置 |
| 5 | 数据长度 | 固定：`0x38`（十进制 56） |
| 6 | 协议常量 | 固定：`0x02` |
| 7 | 页码 | `0x01` = 第一页（按键 0x01-0x09），`0x02` = 第二页（按键 0x0A-0x23） |
| 8 | 主/副功能 | `0x01` = 主功能，`0x02` = 副功能（Fn 层） |
| 9-62 | 按键模块数据 | 9 个按键 × 6 字节 = 54 字节 |
| 63 | 帧尾 | 固定：`0x3F` |
| 64 | CID（重复） | `0xB4` |

> **实测发现与官方文档的差异（byte5-7）：** 壹号本内部文档
> （`GD32--CH32 Mapping analysis26.1.15.xlsx`）将 byte5-7 标注为"保留 `0x00`"。
> 但在 X1 Mini（CH32 MCU）上实测发现 `0x00 0x00 0x00` 完全**不起作用**——MCU 会静默
> 忽略该命令。正确的值为 `0x38 0x02 [页码]`，来源于 HHD 项目。每个字节均经过单独测试：
>
> - **byte5 = `0x38`**：必须的固定常量。使用其他值（0x00、0x36、0x3A、0xFF）
>   均导致命令失败。可能代表数据负载长度（56 字节）。
> - **byte6 = `0x02`**：必须的固定常量。使用其他值（0x00、0x01、0x03）均导致
>   命令失败。并非"总页数"——设为 `0x01` 也不能实现单包更新。
> - **byte7 = 页码**：唯一的可变字段。MCU 根据此值判断当前数据属于哪一页。
>   发送重复页码（如两次 page=1）不会触发更新。

### 第一包（页 1）— 按键 0x01 到 0x09

包含：A、B、X、Y、LB、RB、LT、RT、START

### 第二包（页 2）— 按键 0x0A 到 0x23

包含：BACK/SELECT、L3、R3、十字键上下左右、M1、M2

### 更新机制

> **在 X1 Mini（CH32 MCU）上实测确认：**
>
> - 必须同时发送第一页和第二页才能触发更新。仅发送其中一页会缓存数据，但不会应用。
> - **接收到第二页时触发更新**：当第二页到达且缓冲区中已有第一页数据时，MCU 将两页
>   数据一起应用。
> - **必须先发第一页，再发第二页**。反序发送（先发第二页再发第一页）无法正确触发更新。
> - 发送重复页码（如连发两个 page=1）不会触发更新。
> - 无效页码（如 page=3）会被静默忽略。

### 查询命令

读取当前映射配置时，发送 byte4 = `0x01`（查询），byte8 = `0x01`（主功能）或
`0x02`（副功能）。按键模块区域全部填零。

## 按键模块数据（每按键 6 字节）

```
[物理按键序号] [模式] [映射/键值] [参数1] [参数2] [参数3]
    byte1      byte2    byte3     byte4   byte5   byte6
```

## 模式定义

| 模式 | 名称 | byte3 | byte4 | byte5 | byte6 |
|------|------|-------|-------|-------|-------|
| 0x01 | 手柄模式 | 虚拟按键序号 | 连发：`0x00`=关，`0x01`=半自动，`0x02`=全自动 | 连发按下时间 (10-255ms) | 连发间隔时间 (10-255ms) |
| 0x02 | 键盘模式 | 键盘键值 1 | 键盘键值 2 | 键盘键值 3 | 键盘键值 4 |
| 0x03 | 宏模式 | 宏编号 | 触发方式 | 循环开关 | 循环次数 |
| 0x04 | 全局连发 | 无 (`0x00`) | `0x00`=关，`0x01`=半自动，`0x02`=全自动 | 按下时间 (10-255ms) | 间隔时间 (10-255ms) |
| 0x05 | Fn 键（第二功能） | 无 | 无 | 无 | 无 |

## 各模式详解

### 模式 0x01 — 手柄模式

将物理按键重映射到任意虚拟手柄按键。可选启用局部连发（按住即连发）。

**示例：**

| 说明 | byte1 | byte2 | byte3 | byte4 | byte5 | byte6 |
|------|-------|-------|-------|-------|-------|-------|
| A → A（默认） | 0x01 | 0x01 | 0x01 | 0x00 | 0x00 | 0x00 |
| A → B（重映射） | 0x01 | 0x01 | 0x02 | 0x00 | 0x00 | 0x00 |
| B → X（重映射） | 0x02 | 0x01 | 0x03 | 0x00 | 0x00 | 0x00 |
| B → B 半自动连发 | 0x02 | 0x01 | 0x02 | 0x01 | 0x64 | 0x64 |
| B → X 全自动连发 | 0x02 | 0x01 | 0x03 | 0x02 | 0x64 | 0x64 |
| M1 → LT（出厂默认） | 0x22 | 0x01 | 0x07 | 0x00 | 0x00 | 0x00 |
| M2 → RT（出厂默认） | 0x23 | 0x01 | 0x08 | 0x00 | 0x00 | 0x00 |

- **半自动连发** (`0x01`)：按住按键触发连发。
- **全自动连发** (`0x02`)：单按触发持续连发，再按停止。

### 模式 0x02 — 键盘模式

将物理按键映射为最多 4 个同时按下的键盘按键。

> **键码编码说明：** OXP 固件使用**专有键码编码**，与标准 USB HID Usage Tables 不同。
> 功能键的编码公式为 `F(n) = 0x59 + n`（例如 F1=0x5A，F13=0x66，F14=0x67）。
> 此编码由 srsholmes 的 Apex 固件重映射实现确认。OXP 内部文档未明确记录此编码。

**示例：**

| 说明 | byte1 | byte2 | byte3 | byte4 | byte5 | byte6 |
|------|-------|-------|-------|-------|-------|-------|
| X → 键盘 'A' (0x04) | 0x03 | 0x02 | 0x04 | 0x00 | 0x00 | 0x00 |
| M1 → F14（OXP: 0x67） | 0x22 | 0x02 | 0x01 | 0x67 | 0x00 | 0x00 |
| M2 → F13（OXP: 0x66） | 0x23 | 0x02 | 0x01 | 0x66 | 0x00 | 0x00 |

OXP 专有功能键编码（`F(n) = 0x59 + n`）：

| OXP 键码 | 按键 | OXP 键码 | 按键 | OXP 键码 | 按键 |
|----------|------|----------|------|----------|------|
| 0x5A | F1 | 0x60 | F7 | 0x66 | F13 |
| 0x5B | F2 | 0x61 | F8 | 0x67 | F14 |
| 0x5C | F3 | 0x62 | F9 | 0x68 | F15 |
| 0x5D | F4 | 0x63 | F10 | 0x69 | F16 |
| 0x5E | F5 | 0x64 | F11 | | |
| 0x5F | F6 | 0x65 | F12 | | |

### 模式 0x03 — 宏模式

触发预录制的宏序列。宏系统使用透传宏（单片机 flash 有限，不支持底层保存）。

| 字段 | 说明 |
|------|------|
| byte3 | 宏编号（槽位索引） |
| byte4 | 触发方式 |
| byte5 | 循环开关 |
| byte6 | 循环次数 |

### 模式 0x04 — 全局连发（Turbo）

将按键设为全局连发修饰键。按住此键的同时按其他按键，其他按键会连发。

| 字段 | 说明 |
|------|------|
| byte3 | 无 (`0x00`) |
| byte4 | `0x00` = 关，`0x01` = 半自动，`0x02` = 全自动 |
| byte5 | 按下时间 (10-255ms) |
| byte6 | 间隔时间 (10-255ms) |

**注意**：不要将多个按键设为全局连发键，可能导致异常。

### 模式 0x05 — Fn 键（第二功能键）

将按键设为第二功能修饰键（类似键盘 Fn 键）。按住此键的同时按其他按键，会触发
那个按键的副功能配置。

副功能支持 3 种映射类型：Xbox 按键、键盘按键、触发宏。副功能映射通过包头
byte8 = `0x02` 单独发送配置。

**注意**：不要将多个按键设为 Fn 键。尽量不要映射 BACK 和 START 键，它们是特殊功能键。

## 默认配置

### 第一包（主功能）

```
B4 3F 01 02 38 02 01 01           ← byte5-7: 0x38 0x02 0x01（第一页）
01 01 01 00 00 00    A → A
02 01 02 00 00 00    B → B
03 01 03 00 00 00    X → X
04 01 04 00 00 00    Y → Y
05 01 05 00 00 00    LB → LB
06 01 06 00 00 00    RB → RB
07 01 07 00 00 00    LT → LT
08 01 08 00 00 00    RT → RT
09 01 09 00 00 00    START → START
3F B4
```

### 第二包（主功能）

```
B4 3F 01 02 38 02 02 01           ← byte5-7: 0x38 0x02 0x02（第二页）
0A 01 0A 00 00 00    BACK → BACK
0B 01 0B 00 00 00    L3 → L3
0C 01 0C 00 00 00    R3 → R3
0D 01 0D 00 00 00    十字键上 → 十字键上
0E 01 0E 00 00 00    十字键下 → 十字键下
0F 01 0F 00 00 00    十字键左 → 十字键左
10 01 10 00 00 00    十字键右 → 十字键右
22 01 07 00 00 00    M1 → LT（出厂默认）
23 01 08 00 00 00    M2 → RT（出厂默认）
3F B4
```

注意：出厂默认把 M1 映射为 LT、M2 映射为 RT（手柄模式 `0x01`）。InputPlumber 的
`INIT_CMD_2` 将 M1/M2 改为键盘模式（`0x02`），键码为 F14/F13
（`0x22 0x02 0x01 0x67 0x00 0x00` 为 M1→F14，`0x23 0x02 0x01 0x66 0x00 0x00` 为 M2→F13），
这样可以解除 M1/M2 在 Xbox 手柄上的 LT/RT 镜像，同时为背键分配唯一键码。
OXP 功能键编码公式为 `F(n) = 0x59 + n`（F13=0x66，F14=0x67），
与 USB HID 标准键码（F13=0x68，F14=0x69）不同。

## GD32 与 CH32 的差异

GD32 单片机变体（用于 X1 Pro 370 等早期设备）使用串口通信。协议类似但帧格式不同：

| 特性 | GD32（串口） | CH32（HID） |
|------|-------------|------------|
| 通信方式 | 串口（115200/8/2/偶校验/无流控） | USB HID (VID:1A86 PID:FE00) |
| 命令 ID | `0xF5` | `0xB4` |
| 帧头 | `0xF5 0x3F` | `0xB4 0x3F 0x01` |
| 查询命令 ID | `0xF4` | `0xB4`（byte4=0x01） |
| 按键模块 | 相同的 6 字节格式 | 相同的 6 字节格式 |
| 模式定义 | 相同（0x01-0x05） | 相同（0x01-0x05） |

按键模块数据格式（每按键 6 字节）和模式定义在两个变体之间完全相同。

## InputPlumber 实现说明

我们的 `driver.rs` 中 `INIT_CMD_1` 和 `INIT_CMD_2` 使用正确的包头格式（来源于 HHD 项目）：

```
正确的包头：    B4 3F 01 02 38 20 [page] 01
官方文档包头：  B4 3F 01 02 00 00 00     01  ← 在 X1 Mini 上不工作！
```

壹号本内部文档（`GD32--CH32 Mapping analysis26.1.15.xlsx`）将 byte5-7 标注为
"保留 `0x00`"，但这在 X1 Mini（CH32 MCU）上是**错误的**。在 X1 Mini 上逐字节测试确认：

- `0x38 0x20 [page]` — 有效（InputPlumber 当前使用）
- `0x38 0x02 [page]` — 同样有效（HHD 的 hid_v1.py 使用）
- `0x00 0x00 0x00` — 静默忽略，不会产生任何映射更新

Byte 5（`0x38`）是必须的，可能表示数据负载长度（56 字节）。
Byte 6 接受 `0x02` 和 `0x20`；X1 Mini 固件 readback 返回 `0x20`，这也是 InputPlumber
使用的值。HHD 项目使用 `0x02`，同样可以正常工作。

官方文档可能描述的是 GD32（串口）变体的协议，或者文档本身已过时。CH32（HID）固件
至少要求 byte5 使用 `0x38` 前缀。
